## 1. 需求总览

### 1.1 背景

天文摄影高度依赖天象时机——流星雨有固定的活跃期、行星有冲日窗口、深空目标随季节变化。但用户当前需要分别去查询多个信息源（星图软件、天气预报、天文网站）。

StartTooler 已经聚合了用户的拍摄数据（拍过什么、什么时间拍过），天然可以告诉用户「接下来什么值得拍」。

### 1.2 核心价值

| 维度 | 现在 | 将来 |
|---|---|---|
| 拍摄规划 | 无规划工具 | 日历告诉你好天气 + 好天象何时重叠 |
| 不错过 | 靠社交媒体偶然看到 | 主动提醒关注的星体处于最佳观测期 |
| 月相感知 | 无 | 开机看到今晚月相，决定出摊还是休息 |
| 教育属性 | 无 | 天文知识内嵌，降低新手入门门槛 |

### 2.4 个性化关注

用户可标记「关注星体」：

- 在星图或标签列表中标记星体为「关注」
- 关注的星体在日历中显示距下次最佳观测窗口倒计时
- 深空目标显示最佳观测季节（如「猎户座大星云: 每年12月–2月最佳」）
- 关注的星体进入最佳观测窗口时 → 应用启动时弹出系统通知

### 2.5 拍摄条件评分

根据月相 + 事件信息，给出当晚的综合评分：

```
今晚拍摄评分: ★★★★☆ (4/5)

✅ 月相: 新月 (月龄 1.2 天) — 几乎无月光干扰
✅ 银河季: 银河核心在傍晚后可见
⚠️ 英仙座流星雨: 还有 28 天进入活跃期
```

评分逻辑：
- 月相权重 60%（新月=满分，满月=0 分）
- 活跃事件权重 25%（有流星雨接近极大=加分）
- 季节因素 15%（银河季=加分）

## 3. 数据需求

### 3.1 事件数据文件

本地 TOML 文件（嵌入资源）：`Resources/astronomy-events.toml`

```toml
[[meteor_showers]]
id = "perseids"
name = "英仙座流星雨"
name_en = "Perseids"
active_start = "07-17"
active_end = "08-24"
peak = "08-12"
zhr = 100
parent_body = "109P/Swift-Tuttle"

[[planetary_events]]
type = "opposition"
planet = "mars"
date = "2027-02-19"
visible_china = true

# ... 更多事件
```

> 数据量预估：未来 3 年事件约 200–300 条，TOML 文件 < 50KB，完全可嵌入。

### 3.2 不需要网络

- 日月位置、月相、行星轨道使用 **天文算法（Meeus）的简化实现**
- 事件数据静态嵌入，不依赖在线 API
- 唯一可能需要网络：彗星轨道数据（Phase 2 可选）

### 3.3 用户关注数据

存入 `config.db`：

```
Key: "astronomy.followed_targets"
Value: ["土星", "猎户座大星云", "英仙座流星雨"]
```

## 5. 边界与异常

| 状态 | 处理 |
|---|---|
| 首次打开星历 | 默认显示当月，事件列表已加载 |
| 用户未设置地理位置 | 所有时间使用北京时间（UTC+8），标注「默认时区」 |
| 事件数据过期（超过 3 年） | 日历底部标注「事件数据覆盖至 2028 年」，提示需要应用更新 |
| 无网络 | 一切正常（纯本地数据 + 算法），仅彗星轨道获取不可用 |
| 关注的星体已删除/不存在 | 关注列表同步标签白名单，删除后自动移除 |

## 6. 技术约束

| 项 | 约束 |
|---|---|
| 月相计算 | 使用简化 Meeus 算法，精度 ±1 小时（桌面应用足够） |
| 行星位置 | 仅需冲日/大距日期，不需要实时赤经赤纬 |
| 无 WebView | SkiaSharp 自绘星图 + 月相 |
| 无后台进程 | 不常驻后台，仅在应用启动时检查事件提醒 |

## 8. 不做清单

| 内容 | 理由 |
|---|---|
| 实时星图（类似 Stellarium） | 星图软件竞争激烈，不做重复轮子 |
| 天气集成 | 需要在线 API + 地理定位，可在拍摄计划器（v0.14+）中考虑 |
| 极光预报 | 用户群体以中低纬度为主，需求面窄 |
| 人造卫星过境预报（ISS/星链） | 与天文摄影关联度低 |
| 事件推荐算法（AI 推荐拍什么） | 计划器阶段考虑 |
| 分享到社交媒体 | 非核心需求 |
