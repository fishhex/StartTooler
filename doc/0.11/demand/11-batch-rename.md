## 1. 需求总览

### 1.1 背景

天文摄影的文件名通常来自相机自动编号（`DSC_0001.NEF`、`IMG_4521.CR3`），完全不携带任何语义信息。用户在 Finder/资源管理器中找文件时完全无法判断内容。

已有 AI 打标提供了丰富的结构化元数据（目标名、质量评分等），这些信息天然适合用于自动生成有意义的文件名。

### 1.2 核心价值

| 场景 | 现在 | 将来 |
|---|---|---|
| 找文件 | 打开每张图才能知道拍的是什么 | 文件名直接告诉你目标、参数、日期 |
| 分享给他人 | 发 `DSC_4201.jpg`，对方不知道拍的是啥 | 发 `M31_20260714_ISO800_120s.jpg`，一目了然 |
| 归档 | 相机编号无意义 | 语义化文件名自成目录索引 |
| 多相机混合 | A7R5 的 `DSC_` 和 Z8 的 `_DSC` 混在一起，无法区分 | 统一命名规则，忽略相机差异 |

## 2. 功能需求

### 2.1 重命名模板

提供**预设模板 + 自定义模板**两种方式。

#### 2.1.1 可用变量

| 变量 | 含义 | 示例值 | 来源 |
|---|---|---|---|
| `{target}` | 主目标天体名 | `M31` | AI 标签中频次最高的星体标签 |
| `{tags}` | 所有星体标签（逗号分隔） | `仙女座星系,星系` | AI 标签 |
| `{date}` | 拍摄日期 | `20260714` | EXIF `DateTimeOriginal` |
| `{time}` | 拍摄时间 | `223015` | EXIF `DateTimeOriginal` |
| `{iso}` | ISO 感光度 | `800` | EXIF `ISOSpeedRatings` |
| `{exposure}` | 曝光时长（秒） | `120s` | EXIF `ExposureTime` |
| `{focal}` | 焦距 | `300mm` | EXIF `FocalLength` |
| `{fstop}` | 光圈 | `f2.8` | EXIF `FNumber` |
| `{camera}` | 相机型号缩写 | `A7R5` | EXIF `Model` |
| `{score}` | AI 质量评分 | `85` | `ai_score` |
| `{index}` | 自增序号 | `001` | 按拍摄时间排序 |
| `{orig}` | 原始文件名（不含扩展名） | `DSC_0001` | 现有 `file_name` |
| `{project}` | 项目名 | `2026夏季银河` | 项目目录名 |

#### 2.1.2 预设模板

| 模板名 | 格式 | 示例 |
|---|---|---|
| 标准 | `{target}_{date}_{iso}_{exposure}` | `M31_20260714_800_120s.NEF` |
| 简洁 | `{target}_{date}_{index}` | `M31_20260714_001.NEF` |
| 保留原名 | `{target}_{orig}` | `M31_DSC_0001.NEF` |
| 完整参数 | `{target}_{date}_{time}_{iso}_{exposure}_{focal}` | `M31_20260714_223015_800_120s_300mm.NEF` |
| 评分前缀 | `{score}_{target}_{date}` | `85_M31_20260714.NEF` |

#### 2.1.3 自定义模板

用户可在设置中编辑模板字符串，实时预览效果：

```
模板:  [{target}_{date}_{iso}               ]
预览:  M31_20260714_800.NEF  ← 基于当前选中的第一张照片
```

模板校验：
- 不能为空
- `{` 和 `}` 必须成对
- 未知变量标黄警告但不阻止使用
- 生成的文件名不能包含非法字符（`/` `\` `:` 等，自动替换为 `_`）

### 2.2 操作流程

```
Gallery 多选照片（或单选）
  → 右键「批量重命名…」或工具栏按钮
  → 弹出重命名对话框
  → 选择/编辑模板 → 预览列表 → 确认
  → 执行重命名
  → 完成反馈 + 可选撤销
```

### 2.3 重命名对话框

```
┌─────────────────────────────────────────────────────┐
│  批量重命名 (已选 23 张)                      [×]   │
├─────────────────────────────────────────────────────┤
│  模板: [标准 ▼]  {target}_{date}_{iso}_{exposure}   │
│        预设: [标准] [简洁] [保留原名] [完整参数]     │
│  ─────────────────────────────────────────────────  │
│  预览:                                              │
│  ┌─────────────────────────────────────────────────┐│
│  │ 原文件名            →  新文件名             状态││
│  │ DSC_0001.NEF       →  M31_20260714_800_120s ✅ ││
│  │ DSC_0002.NEF       →  M31_20260714_800_120s ⚠️ ││
│  │ DSC_0003.NEF       →  M31_20260714_800_120s ⚠️ ││
│  │ DSC_0004.CR3       →  M31_20260714_1600_90s  ✅ ││
│  │ IMG_4521.JPG       →  月亮_20260714_200_15s  ✅ ││
│  └─────────────────────────────────────────────────┘│
│                                                     │
│  ⚠️  2 个文件名重复，已自动添加序号后缀             │
│  ⚠️  1 张照片无 AI 标签，目标名为「未标注」         │
│                                                     │
│              [取消]            [确认重命名 23 张]    │
└─────────────────────────────────────────────────────┘
```

### 2.4 冲突与边界处理

| 情况 | 处理 |
|---|---|
| 新文件名重复 | 自动追加 `_{index}` 序号后缀（如 `M31_20260714_800_120s_2.NEF`） |
| 变量值缺失（如无 AI 标签） | `{target}` 填充为「未标注」，预览中以 ⚠️ 标注 |
| EXIF 信息缺失 | 对应变量填充「未知」，如 `{iso}` → `未知ISO` |
| 模板为空 | 禁用确认按钮，提示「请输入模板」 |
| 目标文件已存在（非本次重命名产生的同名文件） | 跳过该文件，预览中标注「已存在」❌ |
| 跨扩展名重命名（.NEF → 改名为 .CR3） | **不允许**，扩展名始终保留原始值 |

### 2.5 重命名执行

- 通过 `System.IO.File.Move()` 物理重命名文件
- 同步更新 `media_files` 表中对应记录的 `file_name` 和 `file_path`
- 同步更新缩略图缓存（按新路径重建缓存 key）
- 更新 OSS 上传队列中对应文件的路径引用
- 全部在一个事务中完成：要么全部成功，要么全部回滚

### 2.6 重命名历史与撤销

| 能力 | 说明 |
|---|---|
| 记录历史 | 每次重命名操作记录在内存中（本次会话内），保留旧名→新名映射 |
| 撤销按钮 | 重命名完成后底部 toast：「已重命名 23 张 [撤销]」（10 秒） |
| 批量撤销 | 点击撤销 → 所有文件恢复原名 → toast「已撤销重命名」 |
| 会话结束后 | 关闭应用后历史清空，无法撤销跨会话的重命名（设计决策） |

## 3. 数据需求

### 3.1 修改 `media_files` 表

无需新表。重命名操作修改已有记录：

```sql
UPDATE media_files SET file_name = @newName, file_path = @newPath WHERE id = @id
```

### 3.2 新增 Repository 方法

```csharp
// IMediaRepository
Task<int> RenameAsync(IReadOnlyList<(string id, string newName, string newPath)> renames, CancellationToken ct);
Task<IReadOnlyList<MediaFile>> GetByFilePathsAsync(IReadOnlyList<string> paths, CancellationToken ct);
```

### 3.3 缓存更新

重命名后需要：
- `ThumbnailService` 迁移或重建缩略图（按新路径存储）
- 如果文件在 OSS 上传队列中，更新队列记录的路径引用

## 4. 与已有功能的关系

| 功能 | 关系 |
|---|---|
| AI 打标 | 标签为空时 `{target}` 降级为「未标注」，提示用户先打标 |
| Gallery 时间轴 | 重命名不改变照片的拍摄日期和排序 |
| OSS 上传队列 | 需同步更新队列中的文件路径 |
| 回收站 | 回收站中的文件**不支持重命名**（操作入口禁用） |
| 搜索 | 搜索建立在 `file_name` 字段上，重命名后搜索自动生效 |

---

## 5. 不做清单

| 内容 | 理由 |
|---|---|
| 实时/自动重命名（导入时自动执行） | 用户对命名偏好差异大，强制自动可能造成困扰 |
| 重命名规则预设共享/导入导出 | MVP 足够，后续可作为设置导入导出的一部分 |
| 基于文件夹结构的重命名 | 归 D08 拍摄会话管理 |
| 支持正则替换 | 模板系统已覆盖主要场景，正则增加学习成本 |
| 跨会话的撤销历史 | 会话结束后历史清空，保持简单 |
